# ClipOps Portable Runs: Run-Dir Portability & Bundling (v0.4)

**Status:** Draft (handoff spec + operational contract)  
**Primary use case:** iOS demo videos (App Store editorial tutorials) rendered by remote agents/runners  
**Secondary use cases:** Web app demos (Playwright), Tauri desktop demos, YouTube clipping pipelines  
**Target repo(s):**
- Producer(s): `cinta` (iOS), future web/desktop producers
- Director/orchestrator: `creativeops/director` (or equivalent wrapper)
- Renderer: `clipper` / ClipOps (Rust)

**Assigned / Owners**
- **ClipOps team (`clipper`)**: maintain v0.4 portability rules + bundling CLI + validations; keep compiled artifacts portable
- **Director team**: treat the run dir as the unit of work; ensure final run dirs are self-contained and renderable on fresh machines
- **Downstream teams (producers: iOS / Web / Tauri)**: emit run-dir-local assets/signals; avoid absolute paths; keep run contract stable

---

## 1) Why this exists

We want a “run directory” (`run_dir/`) to be the **portable, replayable unit** of the whole pipeline:

- a downstream producer records deterministic source media + signals
- a director generates a deterministic plan
- ClipOps validates + compiles + renders

The run dir should be:

- **portable**: zip it, move it to another machine, render it
- **deterministic**: same run dir → same output
- **agent-friendly**: easy to lint, validate, and debug

### 1.1 The core problem

Most pipelines accidentally bake in machine-local paths:

- brand kits pointing at `~/Downloads/.../brand.json`
- fonts pointing at repo-relative or absolute paths
- card images referencing `../../assets/...`

That breaks remote rendering and multi-agent orchestration.

### 1.2 v0.4’s rule

In `clipops.timeline.v0.4`, the plan and compiled artifacts must use **run-dir-relative paths** (and must not escape the run dir).

This is enforced in validation, and supported by a `clipops bundle-run` command that copies fonts into the run dir and rewrites the plan to reference the bundled kit.

## 2) Run dir contract (directory layout)

This is the **recommended canonical layout** (subset is allowed; keep names stable):

```
run_dir/
  inputs/
    *.mp4                  # recorded clips, screen recordings, etc.
    *.wav|*.mp3            # optional voiceovers, music
    *.png|*.jpg            # optional card images, logos
  signals/
    *.json                 # ios_ui_events, camera_focus, words, etc.
    mattes/                # optional matte sequences or videos
  plan/
    timeline.json          # the plan (validated JSON)
  bundle/                  # created by ClipOps tooling (safe to delete/rebuild)
    brand/
      kit.json             # bundled brand kit (rewritten font paths)
      fonts/
        *.ttf|*.otf
  compiled/                # created by `clipops compile` (safe to delete/rebuild)
    segment_map.json
    camera_path.json
    overlay.edl.json
    transition.edl.json
  qa/                      # created by `clipops qa` (safe to delete/rebuild)
    report.json
  renders/
    final.mp4              # output default
    manifest.json          # render manifest (if generated by CLI)
```

### 2.1 What belongs where

- `inputs/`: all heavy assets needed to render (videos/images/audio)
- `signals/`: all facts derived from producers (taps/focus/words/mattes/etc.)
- `plan/`: the declarative edit plan (`timeline.json`)
- `bundle/`: **machine-local → portable** transformation outputs (fonts + rewritten kit)
- `compiled/`: deterministic compile outputs (EDL, segment map, camera path)
- `qa/`: diagnostics and warnings
- `renders/`: final MP4(s)

## 3) v0.4 portability rules (authoring + enforcement)

### 3.1 The “portable path” definition

A path string in v0.4 must be:

1) **relative** (not `/...`, not `~/...`, not `file://...`, not `C:\...`)
2) **run-dir-contained** when resolved and canonicalized (no `..` escape, no symlink escape)
3) **existent** at validation time (for required assets)

### 3.2 Fields that must be run-dir-relative in `clipops.timeline.v0.4`

Enforced by `clipops validate`:

- `plan.brand.kit`
- `plan.assets.*.path` (video/audio/images/mattes)
- `plan.signals.*.path` (ios ui events, words, etc.)
- `card.background.path` when `card.background.type == "image"`

Enforced by brand-kit validation + v0.4 rules:

- `brand.fonts.*.path`

Also enforced indirectly at compile time for v0.4:

- EDL-emitted asset paths (fonts/images/mattes) must be representable as run-dir-relative

### 3.3 What “validate” does (v0.4)

`clipops validate --run-dir <run_dir>`:

- validates `plan/timeline.json` against `schemas/clipops/v0.4/timeline.schema.json`
- resolves and validates paths (v0.4 rules above)
- validates brand kit JSON against `schemas/clipops/v0.4/brand_kit.schema.json`
- validates iOS UI events JSON against `schemas/clipops/v0.4/ios_ui_events.schema.json` for any signal used as `focus_stream`/`pointer_events`

If a path resolves outside the run dir, validation fails with an error like:

- `Path '/abs/...' resolves outside run dir '...' (from 'relative/path.json')`

### 3.4 `lint-paths` is the “cheap early warning”

`clipops lint-paths --run-dir <run_dir>` scans JSON under:

- `run_dir/plan/**/*.json`
- `run_dir/compiled/**/*.json`

and flags strings that look like absolute paths (`/`, `~/`, `file://`, `C:\...`).

This is **not a full security/escape check** (validation is); it’s an agent-friendly quick detector.

## 4) Bundling fonts + brand kit (the portability bridge)

### 4.1 What bundling does

`clipops bundle-run --run-dir <run_dir>` performs these actions:

1) Reads `run_dir/plan/timeline.json`
2) Locates `plan.brand.kit` (supports pre-v0.4 style resolution: plan-relative, run-dir-relative, repo-root-relative)
3) Loads the brand kit and copies referenced font files into:
   - `run_dir/bundle/brand/fonts/`
4) Writes the rewritten brand kit to:
   - `run_dir/bundle/brand/kit.json`
   - with `brand.fonts.*.path` rewritten to `fonts/<file>`
5) Rewrites the plan in place:
   - `plan.brand.kit = "bundle/brand/kit.json"`

Implementation reference:
- `clipops/crates/clipops-core/src/bundle.rs`

### 4.2 What bundling does NOT do (important)

Bundling does **not** automatically copy:

- `inputs/*.mp4` (assumed already in the run dir)
- `signals/*.json` (assumed already in the run dir)
- `card.background` images (must already be in run dir for v0.4 validation)
- arbitrary overlay images or mattes unless they are already run-dir-contained

If your plan references an image outside the run dir, v0.4 validation will still fail.

### 4.3 Side effects and recommended usage

`bundle-run` **modifies `plan/timeline.json`**.

Recommended patterns:

- **Director pipeline**: run bundling on a staging copy of the run dir (safe, clean diffs)
- **Producer pipeline**: keep brand kit references stable; never hardcode machine paths
- **CI / remote runner**: treat `bundle/` as ephemeral; delete + rebuild freely

### 4.4 Expected output (for agents)

The CLI prints a JSON summary (useful for orchestration logs), roughly:

- `brand_kit_src`: resolved source brand kit path
- `brand_kit_dst`: `run_dir/bundle/brand/kit.json`
- `font_copies`: map from font id → copied destination path

## 5) Director workflow (recommended operational sequence)

This is the sequence to make a run dir portable + renderable with high confidence.

### 5.1 “Portable finalize” step (director-owned)

1) Ensure run dir is complete:
   - `inputs/` contains all referenced videos/images/audio
   - `signals/` contains all referenced signals
   - `plan/timeline.json` exists

2) Run bundling:
   - `clipops bundle-run --run-dir <run_dir>`

3) Run fast lint:
   - `clipops lint-paths --run-dir <run_dir>`

4) Run strict validation:
   - `clipops validate --run-dir <run_dir>`
   - If you run from outside `clipper/`, pass `--schema-dir <path-to-schemas/clipops/v0.4>`

5) Compile + QA (optional but recommended before “ship to render farm”):
   - `clipops compile --run-dir <run_dir>`
   - `clipops qa --run-dir <run_dir>`

6) (Optional) Re-run lint-paths after compile:
   - ensures `compiled/` contains no absolute paths

7) Render:
   - `clipops render --run-dir <run_dir> --audio none`
   - (or `--audio copy` if you want to carry source audio + audio clips)

### 5.2 Acceptance criteria (“portable run” DoD)

A run dir is considered portable if:

- `clipops lint-paths` passes (no absolute paths in plan/compiled)
- `clipops validate` passes with schema v0.4
- `clipops compile` passes
- `clipops render` succeeds on a fresh machine where only:
  - the run dir
  - the ClipOps renderer binary + schemas
  are available

## 6) Downstream producer responsibilities (iOS / web / Tauri)

### 6.1 Producers should never emit machine-local paths

Bad:
- `"/abs/path/to/project/inputs/input.mp4"`
- `"~/Downloads/..."`
- `"file:///..."`

Good:
- `"inputs/input.mp4"`
- `"signals/ios_ui_events.json"`

### 6.2 Producers should keep signals and assets stable and localized

Recommended:
- Always place recorded media under `inputs/`
- Always place signals under `signals/`
- If you emit per-clip signals in multi-clip runs, keep them adjacent under `signals/`

### 6.3 When producers create plans

If a producer creates `plan/timeline.json` directly (common early on):

- always target schema `clipops.timeline.v0.4`
- keep `assets.*.path` and `signals.*.path` run-relative
- treat brand kit as an input that will be bundled:
  - set `brand.kit` to a path the director/runner can resolve, then rely on `bundle-run`

## 7) Troubleshooting (common failures and fixes)

### 7.1 `Could not locate schema dir...`

Cause: `clipops validate/compile` tries to find `schemas/clipops/v0.4` by walking up from the current working directory.

Fix:
- run from the `clipper/` repo root, or
- pass `--schema-dir /absolute/path/to/clipper/schemas/clipops/v0.4`

### 7.2 `Absolute paths are not allowed for portable run dirs`

Cause: v0.4 plan references an absolute path in brand kit / assets / signals / card images.

Fix:
- rewrite paths to be run-dir-relative (`inputs/...`, `signals/...`, `bundle/...`)
- use `clipops bundle-run` to rewrite brand kit + fonts into `bundle/`

### 7.3 `Path ... resolves outside run dir ...`

Cause: a relative path escapes the run dir via `..` or symlinks.

Fix:
- move the referenced file into the run dir
- update the plan to a clean run-relative path

### 7.4 Font copy failures in `bundle-run`

Cause: the brand kit references fonts that don’t exist on the machine running bundling.

Fix options:
- check `brand.fonts.*.path` is correct
- vendor fonts in the repo and reference them relative to repo root (bundle-run can resolve via repo root)
- or embed fonts into the run dir upstream and reference them run-relative from the beginning

## 8) Suggested vNext (not required for MVP)

If/when you want to go further than v0.4:

1) **Bundle all external artifacts** (not just fonts):
   - copy any referenced `assets.*.path`, `signals.*.path`, `card.background.path` into run dir
   - rewrite plan accordingly
2) **Ship schemas with the run dir** (optional):
   - `bundle/schemas/clipops/v0.4/*`
   - ensures the run dir is renderable even without a checkout of `clipper`
3) **Produce a single-file archive**:
   - `clipops bundle-run --zip` → `run_dir.bundle.zip`
   - ideal for a render farm / job queue
